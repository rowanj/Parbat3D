View

The View package consists of a group of source files that are responsible for displaying windows, handling events and interfacing with the Windows API. It also handles starting and quiting the application, which includes constructing and deconstructing objects from the other packages as they are required. Each of the main windows have been seperated into their own class, which extends from a general Window class that contains most of the common functionality.

Main

The main.cpp file contains the standard Windows startup execution function that is known as WinMain. This function creates and displays the inital windows and begins to process events through a message loop (see Event Handling). The main file also handles constructing a destroying the main objects that are required when starting the application, quiting the application, loading an image and closing an image.

Windows

The General Window Class

The Window class is resposible for creating the main windows through the Windows API and associating them with their own c++ objects. It also contains code to handle certain events that are common between all of the main windows (see Event Handling). These common events include global key shortcuts, sticking and dragging windows around the screen and destroying windows. Some simple functions are also included to wrap around the Window API.

Main Window

The Main Window is an off-screen window that is not visible to the users. All of the other windows are either directly or indirectly "owned" by this window. Its purpose was to be used as the window that appears in the taskbar, that will automatically minimize and restore the other windows as required. This has now been acheived in a better way, by showing the Overview window in the taskbar instead.

The main purpose of the Main Window now is to hide, show, enable and disable all of the other windows when required.

Overview Window

The Overview Window is considered to be the "main window" of the application according to the user. It is the only window that appears when the program starts up. The job of the overview window is to show an overview of the image that is currently open (if there is one) and to provide the main functions of the program to the user through the use of a menu bar. Apart from the Feature Space Windows, it is the only window that can be minimized to the task bar. Minimizing and restoring this window minimizes and restores the remaining windows that do not appear on the task bar. The overview of the image is displayed using the Display Window class.

Image Window

The Image Window is a resizable window that displays the currently loaded image. It handles navigating and zooming the image by both keyboard shortcuts and scrollbars. The image is displayed inside the window using the Display Window class.

The other main windows are created with the image window set as the "owner". This causes them stay in front of it of the image window, allowing it to be maximized while still being able to access the other windows.

Tools Window

The Tools Window contains multiple tabs that consist of useable tools for the user. It handles creating the tab objects, storing them in a list and showing them one at a time as the user switches between them.

Each of different tabs in the tool window have their own seperate class which is based on a general ToolTab class. This ToolTab class handles constructing a child window for each of the tabs that can be used by the Tool Window.

Preference Window

The Preferences Window contains controls that allow the user to change the preferences. It handles the events associated with these controls and saves them to the settings file.

Contrast Stretch Window

The Constrast Strech Window consists of slider controls and buttons that enable the user to change the contrast of the image that is currently displayed. It handles the events generated from these controls and updates the Overview and Image window displays.

Other GUI Components

Scroll Box

The ScrollBox class is an implementation of a child window that allows the user to scroll its contents vertically. It automatically shows and hides scrollbars as required and moves its own child windows up and down as the user scrolls.

Display Window

The DisplayWindow class is a child window that is used for displaying OpenGL content within a window. This window is used by both the Overview and Image windows as a area to which the OpenGL library can draw to. It also handles certain events associated with the OpenGL display, such as moving and clicking the mouse over the Image window (querying and creating ROIs) and redrawing the display if it is resized.

Sticky Window Manager

The StickyWindowManager class is resposbile for allowing the main windows to snap and move around the screen together. It stores a list of windows that are allowed to snap together and allows them to be registered and unregistered at any time. The Window class calls functions in the StickyWindowManager whenever a window is dragged.

Everytime a window is being dragged by the user, the sticky window manager first checks to see if it is registered. If it is in its list, it then checks to see if it is physically close to any of the other registered windows, or the Overview window. If the window is close to another then it is moved towards it.

When the Overview window is being dragged by the user, the sticky window manager works out which windows are either directly or indirectly connected to it. It then moves the connected windows by the same amount as the Overview window, so that they appear to move with it.

Event Handling

All of the events that are generated by the windows are passed to the application in the form of messages. These messages are retreived by a message loop in the WinMain function and passed to a static processing function that is called a Window Procedure.

The Window class has been written in a way that allows these static Window Procedures to obtain the pointer to a c++ object that is associated with the the window that is receiving the event. This allows the Window Procedures to access the members of a window object when an event occurs.

The default Window Procedure for each of the main windows is set to Window::WindowProcedure. This procedure checks for some events that are common to all windows, such as global shortcut keys and passes any unprocessed message through to the Windows API so that it can perform the default operation.

The Windows API has been written to allow multiple Window Procedures to be chained together. The Window class uses this feature to enable the main windows to handle certain events that they care about, such as a button click, while still being able pass the remaining messages onto the general Window::WindowProcedure. The main windows do this by calling Window::SetWindowProcedure, which returns the address of the previous Window Procedure in the chain.

The majority of the messages that are processed by the windows are still forwarded on to the next procedure in the chain. This is to enable both the general window procedure and the Windows API to handle the messages if it is required.
